## 1. 문제 정의

무인판매대(스마트 진열대) 영상에서 고객의 **구매(집어감)/반환(다시 놓음)** 행동을 감지해 재고와 결제 정보를 CSV로 산출하는 과제.

- 카메라 5대가 같은 진열대를 다른 각도에서 촬영
- 상품 60종 인식
- 평가 기준: 정확도 40점(재고인식) + RTF 20점(처리속도) + 발표 40점 = 60점 만점(정량) + 40점(정성)

**난이도의 핵심 두 가지:**
1. 특정 상품은 진열 위치상 카메라 5대 중 1~2대에서만 물리적으로 보임 (구조적 사각지대)
2. 손이 스쳐가는 순간의 occlusion(가림)·confidence 흔들림을 "진짜 이벤트"와 구분해야 함

## 2. 시스템 아키텍처 (5단계 파이프라인)

```
5개 카메라 영상 → [1] YOLOv7 탐지 → [2] 멀티뷰 퓨전 → [3] 이벤트 감지 → [4] 재고/가격 계산 → [5] CSV 출력
```

| 단계 | 파일 | 역할 |
|---|---|---|
| 탐지 | `src/run_pipeline.py` | 대회 제공 YOLOv7 가중치(`yolov7_custom.pt`, mAP@0.5=98.1%, 약 20만 장 학습). 5카메라를 하나의 batched forward pass로 묶어 GPU 효율화 |
| 퓨전 | `src/multi_view_fusion.py` | 5대 카메라 탐지 결과를 클래스별로 합쳐 "지금 몇 개 보이는가" 산출. 기본은 weighted median(과반 동의 필요) |
| 이벤트 감지 | `src/event_detector.py` | 프레임별 카운트 흔들림을 무시하고 "진짜 재고 변화"만 판정하는 상태 머신. Sliding window median(25프레임) + 연속 확정 프레임(30프레임≈2초) |
| CSV 생성 | `src/csv_generator.py` | 품목명/이벤트번호/구매반환여부/재고수량/총액 |

**RTF 최적화 포인트:** `grab()`/`retrieve()`로 skip 프레임은 H.264 디코딩 자체를 생략, 5카메라 배치 추론으로 forward pass 1회로 축소. 최종 RTF ≈ 0.76 (기준 1.0 이하 → 20점 만점).

## 3. 핵심 난제 3개와 해결 과정

### 3-1. 이벤트 감지 로직 자체의 버그 (Phase 7)

**증상:** 초기 재현율 42%. 반환 이벤트 재현율 7% vs 구매 76% — 극단적 비대칭.

**원인 2가지:**
- **UNKNOWN 상태 버그**: 초기 재고 추정에 없던 클래스가 UNKNOWN 상태로 시작 → 첫 감지(반환 이벤트)가 발생하면 그 값을 "원래 초기 재고"로 잘못 확정해버림. → UNKNOWN 상태 자체를 제거, 전부 STABLE(committed=0)로 시작하도록 재설계.
- **연쇄 잠김 버그**: 반환 감지가 한 번 실패하면 committed 값이 안 갱신돼 "선반이 비어있는데 구매?"라는 제약에 걸려 다음 구매 이벤트까지 도미노처럼 차단됨. → 구매-차단 제약 제거 + 이벤트 발생 후 sliding window 히스토리 리셋(쿨다운 효과).

**결과:** 재현율 42% → 79% (44/105 → 83/105)

### 3-2. 멀티뷰 퓨전의 구조적 사각지대 (Phase 7~10)

**증상:** `bumblebee_albacore`, `dove_white/pink`, `redbull`, `spam` 등 특정 상품이 영상 전체에서 count=0. 모델 자체는 conf 0.5~0.9로 잘 탐지하는데도 최종 출력에는 전혀 안 잡힘.

**진단:** `tools/probe_low_confidence.py`로 conf=0.05까지 낮춰 재탐지한 결과, 이 상품들은 **5대 카메라 중 3대 이상이 동시에 본 적이 물리적으로 단 한 번도 없음**(최대 1~2대). weighted median 퓨전은 과반(3/5) 동의를 요구하므로 median이 구조적으로 항상 0이 됨. 학습 데이터 부족 문제가 아님(더 적은 데이터의 다른 클래스는 정상 인식됨을 대조 확인).

**해결:** `CLASS_QUORUM_OVERRIDE` — 이 클래스들만 필요 동의 카메라 수를 1~2대로 낮춤. 단순히 1로 낮추면 카메라 1대의 노이즈에도 즉시 이벤트가 확정돼 중복발화 발생 → 클래스마다 quorum=1 vs 2를 실측 비교해 절충점을 찾음 (예: `dove_white`는 quorum=1일 때 노이즈로 중복발화 4건 → quorum=2로 1건까지 감소).

### 3-3. 카메라별 occlusion 인지 가중치 (Phase 16~18) + 이벤트 발화 타이밍 정밀 튜닝 (Phase 29~30)

**증상:** quorum 조정만으로 해결 안 되는 클래스가 남음. `haribo`, `milano` 등은 손이나 다른 상품에 가려질 때만 일부 카메라 신호가 사라지는 패턴(항상 안 보이는 게 아니라 간헐적).

**해결 — 동적 카메라 가중치:** 클래스별로 "지금 이 카메라가 가려져 있는가"를 그 클래스 자신의 confidence만으로 실시간 판단해 투표권을 낮추는 방식(`compute_per_class_cam_weights`) 도입. 초기 재고 추정 단계에도 동일 적용 → FP(가짜 이벤트) 완전히 0 달성한 부수 효과까지 얻음.

**추가 해결 — per-class 카메라 화이트리스트 (Phase 24):** `campbells`가 유사 상품(`campbells_chunky`)과 계속 혼동. per-camera 원시 로그 분석 결과 특정 카메라(cam4)가 혼동 주범 → `CLASS_CAM_WHITELIST`로 해당 클래스에 대해 신뢰 카메라만 남김. Order F1 96.6%→98.6%.

**최종 마무리 — LCS 역추적으로 순서 역전 진단 (Phase 29~30):** Count/Time은 100%인데 Order만 99%에 머묾. 자체 제작한 LCS 역추적 스크립트로 "정확히 어느 이벤트 쌍이 뒤바뀌었는지" 특정 — `campbells`(GT 11초)와 `monster_energy`(GT 12초)가 실제로는 1초 차이인데 파이프라인 순서가 역전됨. 원인 추적 결과 campbells는 메인 루프에서 계속 다른 클래스로 오분류되고 있었음(모델 레벨 한계). `per_class_confirm`(클래스별 이벤트 확정 대기 프레임)의 발화 시각을 5개 지점 실측 → "early zone"/"late zone"이라는 불연속 두 구간을 발견하고 이를 근거로 역산 → **Order F1 99%→100%** 달성.

## 4. 정량 평가 방법론 자체를 만든 이야기

단일 지표만 보면 오판하기 쉬움을 여러 번 겪음:
- 클래스별 개수(count)만 맞으면 순서가 틀려도 "맞다"고 카운트됨
- 반대로 전체 시퀀스 정렬(order) 하나가 무관한 다른 클래스의 판정까지 깨뜨림

→ 3가지 지표를 동시에 보는 자체 채점 도구(`tools/score_methods.py`) 제작:

| 방법 | 측정 대상 |
|---|---|
| **count** | (클래스, 액션) 빈도 비교 |
| **order** | 전체 이벤트 시퀀스의 LCS(최장 공통 부분열) — 순서만 봄 |
| **time** | 실제 발화 시각을 GT와 직접 비교. 시스템 고유 지연(CONFIRM_FRAMES로 인한 median offset, 약 +2.4초)을 자동 추정해 보정 후 비교 |

**GT 데이터 자체도 재검수 대상이었음:** `ground_truth_v2.csv` 최초 버전은 시간 단위가 60배 부풀려진 버그가 있었고(예: 239초짜리 영상에 "233분" 이벤트), 순서가 일부 뒤바뀐 행, 클래스명이 잘못 기록된 행 등도 발견해 팀원이 직접 재검수·수정. 이후 이전에 "모델 문제"로 오판했던 몇몇 케이스가 사실 "GT 기록 오류"였음이 드러남.

## 5. 최신 논문 기법 적용 시도 및 실패 분석

> 이 섹션은 "최신 연구 흐름을 이해하고 있으며, 왜 이 특정 문제/데이터에는 안 맞는지 근거를 갖고 판단했다"는 것을 보여주는 파트. 단순히 성공 사례만 나열하는 것보다 설득력 있음.

| 기법 | 출처 | 아이디어 | 결과 | 실패/기각 원인 |
|---|---|---|---|---|
| **ByteTrack** | ECCV 2022 (arxiv 2110.06864) | low-confidence 탐지도 버리지 않고 2-stage로 기존 트랙에 매칭 | 로컬 검증 긍정적(FP track 억제 확인) | 서버 정량 검증까지는 미완료 상태로 문서화 |
| **SeqNMS** | 2016 (arxiv 1602.08465) | 연속 프레임에 안 나타나는 탐지는 노이즈로 간주해 억제 | Count/Order 모두 하락 | sparse하지만 실제인 감지(카메라 1~2대만 보는 상품)까지 같이 억제됨 |
| **EMA Smoothing** | — | sliding median 대신 지수이동평균으로 반응성 개선 | Count/Order 모두 하락 | 이벤트 발생 후 내부값 리셋이 안 돼 두 번째 이벤트가 억제됨 |
| **Ghost Detector** | ICCV 2023, "Objects Do Not Disappear" (arxiv 2308.04770) | 일시적 미탐지 시 마지막 위치에 가상 탐지 삽입 | Count/Order 큰 폭 하락 | "일시적 미탐지"와 "실제 이벤트 발생"을 구분 못 함 — 진짜 이벤트 순간에 ghost가 오히려 더 오래 유지되며 지연·오탐 발생 |
| **WBF 앙상블** (YOLOv7 + YOLO11 KD) | — | 두 모델이 서로 놓치는 프레임을 상호 보완 | FP 대거 발생 | 두 모델의 confidence calibration이 달라 동일 threshold가 다르게 작동. 앙상블은 "비슷한 수준으로 튜닝된 두 모델"이 전제인데 조건 불충족 |
| **YOLO11 / YOLO11 KD 재학습** | Ultralytics | 최신 아키텍처로 백본 교체 | Order F1 73~91% (기존 대비 -7~25%p) | 대회 제공 가중치는 약 20만 장 학습, 자체 학습은 1.8만 장 — 데이터 규모 10배 이상 차이. mAP는 비슷해도 파이프라인 정확도 격차 큼 |
| **Hand-Object Interaction Detection** | 2025 (arxiv 2507.13326) | count 변화 대신 "손이 물건을 집는 동작"을 직접 감지 | 적용 안 함(구조적 불가) | egocentric(1인칭) 카메라 전용 논문 — 고정 카메라 시점과 전혀 다름, 학습 데이터도 없음 |
| **Multi-camera 3D/BEV Fusion** | MCBLT, 2024 (arxiv 2412.00692) | 5캠을 homography로 Bird's Eye View 통합 | 적용 안 함(구조적 불가) | 카메라 내/외부 캘리브레이션 행렬이 대회 환경에 제공되지 않음 |

**시사점 (발표 멘트 예시):** "최신 기법일수록 무조건 좋은 것이 아니라, 우리 데이터의 특성(카메라 사각지대, sparse한 detection, 캘리브레이션 부재)과 맞는지가 중요했다. 특히 시간 축 스무딩 계열(SeqNMS, EMA)은 공통적으로 '실제로 드물게 보이는 신호'와 '노이즈'를 구분하지 못해 실패했는데, 이는 이 문제의 근본적 어려움이 감지 자체보다 노이즈-신호 구분에 있다는 방증이기도 하다."

## 6. 최종 성과

### 정량 결과 (main, Phase 30 기준, 2026-07-01)

| 항목 | 값 |
|---|---|
| Count F1 | **100%** |
| Order F1 | **100%** |
| Time F1 (±3초 보정) | **100%** |
| RTF | **~0.76** (기준 1.0 이하 → 20점 만점) |
| 추정 총점 (정확도+RTF, /60) | **약 60점** |
| 모델 mAP@0.5 | 98.1% (대회 제공 원본 가중치 사용) |

### 범용 파이프라인 대비 기여도 (발표 핵심 근거)

영상 특화 튜닝(quorum override, 카메라 화이트리스트, per-class confirm 등)을 전부 제거한 "완전 범용 파이프라인"을 별도로 측정해 튜닝의 기여도를 정량화함(`test/generic-pipeline` 브랜치, Phase 25):

| 지표 | 범용 파이프라인 | 최종(튜닝) | 차이 |
|---|---|---|---|
| Count F1 | 93.6% | 100% | +6.4%p |
| Order F1 | 91.7% | 100% | +8.3%p |
| Time F1 | 89.9% | 100% | +10.1%p |
| 추정 총점 | ~56.7점 | ~60점 | +3.3점 |

→ **"기본 파이프라인만으로도 상당한 성능(order 91.7%)이 나오지만, 이 특정 진열대/카메라 배치를 정밀 분석해 튜닝함으로써 완벽에 도달했다"**는 스토리로 발표 가능.

## 7. (부록) 상세 실험 로그 — 슬라이드 백업용

### 브랜치별 A/B 테스트 요약 (일부)

| 브랜치/시도 | 변경 내용 | 결과 | 판정 |
|---|---|---|---|
| tracker max_age 3→15 | SORT 트래커 파라미터 | count +0.9%p, order 동급 | ✅ 채택 |
| camera-weights (per-class occlusion) | 카메라별 동적 가중치 | order 85.4%→91.0%(1단계), haribo 완전 해결 | ✅ 채택 |
| per-class 카메라 화이트리스트 | campbells/milano/dove_white | order 96.6%→98.6% | ✅ 채택 |
| milano confirm=45/90/150 등 다수 | milano 타이밍 조정 | 전부 과다발화 또는 순서 악화 | ❌ 기각, 최종은 whitelist로 해결 |
| campbells quorum=1 | 카메라 완화 | cam4 혼동으로 FP 3배 폭증 | ❌ 기각 |
| init_frames 30→60 | 초기재고 추정 윈도우 확대 | 일부 클래스 개선, 다른 클래스(quaker) FP 회귀 | ❌ 기각 |
| SeqNMS / EMA / Ghost Detector | 논문 기반 시간축 처리 | 전부 성능 하락 | ❌ 기각 (5장 참고) |

### 핵심 확정 파라미터 (main, `src/event_detector.py` / `src/multi_view_fusion.py`)

```
WINDOW_SIZE = 25          # median 슬라이딩 윈도우
```
