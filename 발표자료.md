# REDRED 발표 자료 (v3 — 여정 순서 서사 구조)

> 무인판매대 상품인식 시스템 경진대회 — 정성 발표용 정리
> 팀명: REDRED | 팀원: 조강희 / 정현수 / 박준영 (서울대 전기정보공학부)
> 최종 갱신: 2026-07-09

이 문서는 실제 문제 해결 **여정 순서**대로 재구성한 버전. "YOLOv7으로 100% → 그게 왜 RF-DETR+SAM2로 이어졌는지"를 겪은 순서 그대로 보여준다: 파이프라인 개선 → 다른 모델(YOLO11) 시도 실패 → 원인 규명(raw data 양) → 논문 리서치 → RF-DETR 채택 → 이식 실패 → 진단(raw data 부족) → SAM2로 데이터 증대 → 재설계 → 100% 재현 → 비교.

---

## 목차 (여정 순서)

0. 문제 정의 / 평가 기준 + 배경지식(3종 채점, 리더보드)
1. YOLOv7 — 자체 파인튜닝 실패 → 파이프라인 개선으로 방향 전환
2. YOLOv7 — 4단 레이어 파이프라인 상세 → 100% 달성
3. YOLO11 시도 → "mAP는 높은데 왜 반영이 안 되지?" → raw data 양이 진짜 원인이라는 깨달음
4. 논문 · 업계 리서치 → 이태호 교수 추천 기법(ByteTrack 등) 실패 → RF-DETR 채택
5. RF-DETR 전체 스택 도입(전처리~판단) → 이식만으로는 그냥 안 나옴
6. 특정 클래스 오류 진단 → "raw data를 더 만드는 것"이 근본 해결책
7. SAM2로 raw data 증대 + 재설계 → YOLOv7보다 얕은 파이프라인으로 100% 재현
8. RF-DETR의 confidence 캘리브레이션 우위 → 하드코딩 최소화 + RTF
9. 두 아키텍처 비교
10. 결론

## 0. 문제 정의 / 평가 기준 + 배경지식

무인판매대(스마트 진열대) 영상에서 고객의 **구매(집어감)/반환(다시 놓음)** 행동을 감지해 재고와 결제 정보를 CSV로 산출하는 과제. 카메라 5대가 같은 진열대를 다른 각도에서 촬영, 상품 60종 인식.

**평가:** 재고인식 정확도 40점 + RTF(처리속도) 20점 + 정성평가·발표 40점 = **100점**

**배경지식 — 이 프로젝트를 관통하는 도구들 (0부 방법론):**

| 도구 | 역할 |
|---|---|
| **3종 채점 방법론** (count/order/time, `tools/score_methods.py`) | 대회가 공식 채점 기준을 공개하지 않는 상황에서 직접 설계. 개수만 맞으면 순서가 틀려도 맞다고 카운트되는 등 단일 지표의 맹점을 서로 보완 |
| **LCS 역추적 알고리즘** | Order F1 미스매치가 발생했을 때 "정확히 어느 이벤트 쌍이 순서를 깨뜨렸는지"를 짚어내는 자체 진단 스크립트 |
| **실험노트형 디버깅 히스토리** (`output/leaderboard.html`) | 매 실행의 계기/문제/다음단계를 펼쳐볼 수 있는 기능. 55회 이상의 실행 전부 기록 |
| **End-to-end 자동화** (`run_test.sh`) | 파이프라인 실행 → 채점(3종) → 리더보드 갱신 → git push까지 한 번에 연결 |
| **`debug_frame_counts.csv` 직접 분석 + 로컬 replay 도구** | 서버 없이 로컬에서 프레임별 raw count를 재생해 "모델 미검출"과 "로직 버그"를 구분 |
| **candidate 시각 역산** (`fire_time` 공식) | 이벤트 확정 시각을 역산해 `confirm_frames`를 정밀 튜닝 — YOLOv7과 RF-DETR 양쪽에 동일 기법 재사용 |

이 도구들이 앞으로 나올 모든 진단·튜닝 과정에서 반복적으로 등장한다.

## 1. YOLOv7 — 자체 파인튜닝 실패 → 파이프라인 개선으로 방향 전환

**출발점:** 대회 제공 가중치(`yolov7_custom.pt`, mAP@0.5 = 98.1%, 약 20만 장으로 학습).

**첫 번째 시도 — 모델 자체를 개선해보자:** Cut&Paste 증강 5,000장을 추가해 30epoch 자체 파인튜닝.
→ **결과: mAP@0.5 0.9904 → 0.9840로 오히려 하락.** 증강 이미지의 조명·그림자·배치 분포가 실제 테스트 영상과 괴리돼 모델이 오히려 그 분포에 맞춰지며 성능이 저하된 것으로 추정.

**방향 전환:** "모델 자체를 손대는 건 위험하다 — 대회 제공 가중치는 그대로 두고, 그 위에서 돌아가는 **파이프라인 쪽을 개선하자**."

## 2. YOLOv7 — 4단 레이어 파이프라인 상세 → 100% 달성

**속도 최적화 먼저:** `grab()`/`retrieve()`로 skip 대상 프레임은 H.264 디코딩 자체를 생략, 5카메라를 하나의 batched forward pass로 묶어 GPU 추론 1회로 축소. 기록값 RTF ≈ 0.763(2026-07-01).

파이프라인은 클래스별 물리적 특성에 맞춘 **4단 레이어**로 구성:

### 레이어 1 — 멀티카메라 퓨전 (Quorum + 화이트리스트)
- 기본은 weighted median(과반 동의)
- `CLASS_QUORUM_OVERRIDE`: 일부 상품은 진열 위치상 카메라 5대 중 1~2대에서만 물리적으로 보임(구조적 사각지대) — 클래스별로 필요 동의 카메라 수를 1~2대로 완화 (8개 클래스 적용)
- `CLASS_CAM_WHITELIST`: `campbells`/`milano`/`dove_white` 등 특정 카메라가 유사 상품과 계속 혼동을 일으키는 걸 발견 → 그 카메라를 신뢰 목록에서 원천 배제

### 레이어 2 — Occlusion-aware 카메라 가중치
- 손이나 다른 상품에 가려질 때 "일시적 가림"과 "실제 재고 변화"를 구분해야 함
- 전역 평균 → 클래스별 판단 → "카메라 i가 0인데 나머지 4대 중 N=2대 이상 양수면 i를 가려진 것으로 보고 제외" 개별 카메라 단위로 3단계 정교화
- 클래스 자신의 confidence만 사용(전역 평균 아님) → Occlusion 감지율 8%→27~38%로 급증, FP(가짜 이벤트) 완전 0 달성

### 레이어 3 — SORT 트래커
- `max_age=15` — occlusion gap(실측 0.4~1.5초)을 버티는 짧은 id 유지
- RTF는 거의 공짜(GPU 추론 비용 없이 CPU Kalman/IoU 매칭만 추가)

### 레이어 4 — Hysteresis 상태머신 + per-class confirm
- STABLE/CANDIDATE 2-state 머신, `WINDOW_SIZE=25`(sliding median), `CONFIRM_FRAMES=30`(연속 확정 프레임, ~2초)
- 클래스별·**방향별**(구매/반환) `per_class_confirm` 개별 튜닝으로 이벤트 확정 시각을 정밀 조정

### 최종 성능
**Count F1 100% / Order F1 100% / Time F1 100%, RTF ≈ 0.763**

## 3. YOLO11 시도 → "mAP는 높은데 왜 반영이 안 되지?"

**가설:** 대회 가중치(YOLOv7)보다 최신·경량 아키텍처(YOLO11)를 쓰면 정확도와 속도 모두 개선되지 않을까?

**실행:** 증강 제외 18,392장으로 YOLO11m 50epoch 재학습.
**결과:** mAP 98.4%로 대회 가중치(98.1%)와 유사하거나 오히려 살짝 높음. **그런데 파이프라인 order F1은 73.4%로 -23.2%p 폭락.** KD(Knowledge Distillation)까지 적용해도 91.1%(YOLOv7 대비 -7.5%p)로 못 따라옴.

**"mAP는 높은데 왜 실제 영상에서는 반영이 안 되지?"** — 이 의문을 파고든 결과, 진짜 원인은 모델 구조가 아니라 **raw data 양**이었다:
- 대회 제공 YOLOv7 가중치는 **약 20만 장**으로 학습된 것
- 팀이 자체 재학습에 쓸 수 있던 데이터는 **1.8만 장**뿐 (10배 이상 격차)
- mAP는 표준 벤치마크에서 비슷하게 나와도, 실제 테스트 영상의 다양한 각도·조명·occlusion 조합을 커버하려면 학습 데이터의 절대량이 훨씬 더 중요하다는 것을 확인

**이 깨달음이 이후 전략을 결정했다:** "그럼 다음에 다른 모델을 시도할 땐, 모델 구조만 바꾸는 게 아니라 raw data 자체를 어떻게 확보할지가 핵심이다."

## 4. 논문 · 업계 리서치 → 이태호 교수 추천 기법 실패 → RF-DETR 채택

YOLOv7 파이프라인이 100%에 도달한 뒤에도 남는 질문 — "이 결과가 YOLOv7이라는 특정 모델에만 통하는 우연은 아닐까?"에 답하기 위해 최신 논문·업계 동향을 조사했다.

**이태호 교수 추천 기법(ByteTrack 등) 시도 → 실패:**
- ByteTrack(2-stage matching, 저신뢰도 detection도 track 유지) 실제 구현·A/B 테스트
- 결과: count 93.6% / order 91.7% / time 89.9%로 오히려 **악화**
- 원인: 저신뢰도 detection을 계속 유지하는 방식이 신규 track 생성을 차단하거나 과다발화를 유발 — 이 프로젝트의 occlusion 패턴과는 안 맞음

이 외에도 SeqNMS/EMA Smoothing/Ghost Detector/WBF 앙상블/BEV 3D fusion 등 시간축·앙상블 계열 기법을 추가로 검토·구현했으나 전부 기각(6장 표 참고).

**결론:** 시간축 보정 계열로는 한계가 뚜렷했다. **아예 다른 탐지기 아키텍처**로 넘어가기로 결정 — **RF-DETR**(Transformer 기반, NMS 불필요, confidence 캘리브레이션이 상대적으로 우수하다고 알려진 모델) 채택.

## 5. RF-DETR 전체 스택 도입 → 이식만으로는 그냥 안 나온다

별도 브랜치(`feat/rfdetr-sam2`)에서 진행, main·최종 제출본은 건드리지 않는 병행 실험. RF-DETR을 도입하되, **전처리부터 판단까지 스택 전체**를 새로 구성해야 했다.

**1차 시도 — YOLOv7 코드를 그대로 이식:** 기존 `event_detector.py`/`multi_view_fusion.py`를 그대로 가져다 쓴 `run_pipeline_rfdetr.py`로 첫 채점.
- 10epoch order F1 85.2%
- 50epoch 재학습(보강 없음) 후엔 오히려 **82.6%로 악화** — 과다발화 9종 신규 발생. mAP 상승이 파이프라인 성능으로 이어지지 않는 패턴을 이 프로젝트에서 두 번째로 확인(YOLO11 때와 동일한 교훈)
- RTF 안전권 확보를 위해 `skip=3`/`conf=0.35`로 재조정, `CLASS_CONF_OVERRIDE` + per-class whitelist/quorum 등 YOLOv7 세션과 유사한 방식으로 누적 적용해 order F1 85.6%까지는 개선

**"YOLOv7의 진단법과 튜닝 스택을 그대로 옮기는 것만으로는 100%에 못 미친다"** — 이 시점에서 두 가지가 동시에 필요하다는 게 분명해졌다: (1) 데이터 자체를 늘려야 한다, (2) 파이프라인도 RF-DETR 특성에 맞게 다시 설계해야 한다.

## 6. 특정 클래스 오류 진단 → "raw data를 더 만드는 것"이 근본 해결책

85.6%에서 정체된 원인을 클래스별로 뜯어봤다. `campbells`/`hunts_sauce`/`cheerios` 등 특정 클래스가 구조적으로 계속 잘 안 잡혔다.

**진단:** YOLO11 때와 같은 패턴 — 이 클래스들은 모델 구조나 파이프라인 로직의 문제가 아니라, **학습 데이터에서 해당 클래스의 노출 빈도 자체가 부족**해서 생기는 문제였다.

**결론:** 파라미터를 더 손보는 대신, **raw data를 늘리는 쪽이 근본적인 해결책**이라고 판단 — 대회가 제공한 건 20만 장으로 학습된 **가중치뿐**이고 원본 이미지는 팀에 전달된 적이 없었으므로, 데이터를 늘리려면 팀이 직접 확보해야 했다.

## 7. SAM2로 raw data 증대 + 재설계 → YOLOv7보다 얕은 파이프라인으로 100% 재현

### 데이터 증대
- 팀이 보유한 학습셋(2만여 장) 전량을 COCO 포맷으로 변환(`tools/yolo_to_coco.py`)
- **SAM2**로 대회 테스트 영상 자체를 프레임화해 도메인 데이터를 직접 추가 추출·라벨링(`tools/sam2_video_label.py`) — 학습-테스트 분포 차이를 줄임
- 약한 클래스(`campbells`/`hunts_sauce`/`cheerios` 등)는 SAM2 캡처 문턱을 **0.3→0.1**로 낮춰 추출 강화
- 표적 클래스가 포함된 이미지를 **5배 오버샘플링**(선별이 아니라 중복 노출)
- 1차(보강 없음) mAP 0.898 → 2차(보강 적용) **mAP 0.9045**(@0.5:0.95 — YOLOv7의 @0.5 mAP와 기준이 달라 직접 비교 불가)

### 파이프라인 재설계 (`src/rfdetr_native_pipeline.py`)
YOLOv7의 4단 레이어를 그대로 이식하는 대신, RF-DETR 특성에 맞춰 **처음부터 다시 설계**:
- **Binary presence** — 카운트 개념 자체를 제거(GT 재고는 항상 0/1)
- **Noisy-OR 확률 융합**(`P = 1-∏(1-conf_i)`) — YOLOv7의 quorum+화이트리스트+가중치 4단 스택을 **확률식 하나로 대체**
- **Adaptive confirm** — 신호 강도에 따라 확정 프레임 수 자동 보간, 방향별(구매/반환) override 지원
- **좌표 하드코딩 완전 제거** — `relabel_regions`(임시 교정 메커니즘)를 화이트리스트 튜닝으로 대체 후 코드에서 삭제

### 정밀 튜닝 (round1~11)
YOLOv7 세션과 동일한 방법론(candidate 시각 역산 + 매 수정 전 전체 시퀀스 재시뮬레이션)으로 order F1 92.4%→100%. whack-a-mole 순서충돌 반복 해결, 방향별 confirm override 신규 도입(macadamia 등), hunts_sauce 방향별 재검증까지 거쳐 **Time F1도 100%** 달성.

### 최종 성능
**Count F1 100% / Order F1 100% / Time F1 100%** — YOLOv7의 4단 레이어보다 **더 얕은 구조**(binary presence + 확률식 하나 + adaptive confirm)로 동일한 만점 재현.

## 8. RF-DETR의 confidence 캘리브레이션 우위 → 하드코딩 최소화 + RTF

**왜 RF-DETR 쪽 파이프라인이 더 얕아도 됐는가:** confidence calibration 차이 때문이다.

- **Calibration이란:** 모델이 "confidence 0.9"라고 낸 예측들을 모아봤을 때 실제로 약 90%가 맞아야 "잘 캘리브레이션된" 모델. 안 그러면 confidence 숫자를 확률로 그대로 못 믿는다.
- **YOLOv7**: confidence를 있는 그대로 못 믿어서 quorum·화이트리스트·occlusion가중치·트래커라는 **우회 장치를 4단으로 겹겹이** 쌓아야 했다.
- **RF-DETR**(Transformer 기반): confidence가 상대적으로 더 잘 캘리브레이션돼 있어서, 5개 카메라의 confidence를 **확률식(Noisy-OR) 하나로 곧바로 결합**할 수 있었다.

**RTF는 정직하게 다룬다** — 절대값(RF-DETR 기록값 0.6856)은 이 세션에서 재현되지 않아 미검증이지만, 2026-07-09 같은 서버·같은 시간대 나란히 재측정한 상대 비교에서는 **RF-DETR(0.905~0.914)이 YOLOv7(0.933~0.949)보다 매번 더 빨랐다**. 두 아키텍처 모두 대회 기준(RTF ≤1 → 만점)은 여유 있게 통과.

## 9. 두 아키텍처 비교

| 항목 | YOLOv7 | RF-DETR + SAM2 |
|---|---|---|
| 탐지기 가중치 | 대회 제공 (20만 장 사전학습) | 팀 보유 2만여 장 + SAM2 도메인 데이터, mAP 0.898→0.9045 |

### 시도했으나 기각한 것들 (전체)
