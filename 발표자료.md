## 0. 문제 정의 (도입부)

무인판매대(스마트 진열대) 영상에서 고객의 **구매(집어감)/반환(다시 놓음)** 행동을 감지해 재고와 결제 정보를 CSV로 산출하는 과제.

- 카메라 5대가 같은 진열대를 다른 각도에서 촬영
- 상품 60종 인식
- 평가: 정확도 40점(재고인식) + RTF 20점(처리속도) + **정성평가 40점**

**난이도의 핵심 두 가지:**
1. 특정 상품은 진열 위치상 카메라 5대 중 1~2대에서만 물리적으로 보임 (구조적 사각지대)
2. 손이 스쳐가는 순간의 occlusion(가림)·confidence 흔들림을 "진짜 이벤트"와 구분해야 함

## 1. AI 모델 구조 설계

**기본 선택: 대회 제공 YOLOv7 가중치(`yolov7_custom.pt`) 유지.**
- mAP@0.5 = 98.1%, 약 20만 장으로 사전 학습된 가중치
- `attempt_load`의 `attempt_download`가 경로를 소문자로 바꿔버려 Linux(대소문자 구분 파일시스템)에서 파일을 못 찾는 문제 발견 → `torch.load` 직접 사용으로 우회 (`src/run_pipeline.py`의 `load_model()`)
- PyTorch 1.12 호환성 패치: `Upsample.recompute_scale_factor = None`

**대안 아키텍처를 실제로 검증하고 근거를 갖고 기각:**

| 시도 | 학습 조건 | 결과 | 판단 |
|---|---|---|---|
| YOLO11m 재학습 | 증강 제외 18,392장, 50epoch | mAP 98.4%(원본과 유사), 그러나 파이프라인 order F1 73.4%(-23.2%p) | ❌ mAP는 비슷해도 실제 이벤트 검출력이 크게 낮음 |
| YOLO11m + Knowledge Distillation | 위와 동일 데이터 | order F1 91.1%(순정 YOLO11보다 개선, 그러나 YOLOv7 대비 -7.5%p) | ❌ KD로도 데이터량 격차를 못 메움 |
| YOLOv7 자체 파인튜닝(30 epoch) | Cut&Paste 증강 5,000장 추가 | mAP@0.5 0.9904→0.9840로 오히려 하락 | ❌ 증강 분포가 실제 테스트 영상과 괴리, 미채택 |

**결론(발표 포인트):** 최신/더 가벼운 아키텍처(YOLO11, RTF 0.47로 YOLOv7의 0.75보다 빠름)를 시도했지만, **대회 제공 가중치는 20만 장 학습인데 우리 자체 학습은 1.8만 장**으로 10배 이상 데이터 격차가 있어 mAP는 비슷해도 실제 파이프라인 정확도에서 크게 밀림 — "무조건 최신 모델이 정답이 아니라, 데이터 규모가 확보된 모델을 쓰는 게 이 과제에서는 더 합리적"이라는 근거 있는 설계 판단.

**시스템 전체 구조 (5단계 파이프라인):**

```
5개 카메라 영상 → [1] YOLOv7 탐지 → [2] 멀티뷰 퓨전 → [3] 이벤트 감지 → [4] 재고/가격 계산 → [5] CSV 출력
```

| 단계 | 파일 | 역할 |
|---|---|---|
| 탐지 | `src/run_pipeline.py` | 5카메라를 하나의 batched forward pass로 묶어 GPU 효율화 |
| 퓨전 | `src/multi_view_fusion.py` | 5대 카메라 탐지 결과를 클래스별로 합쳐 "지금 몇 개 보이는가" 산출 |
| 이벤트 감지 | `src/event_detector.py` | 프레임별 카운트 흔들림을 무시하고 "진짜 재고 변화"만 판정하는 상태 머신 |
| CSV 생성 | `src/csv_generator.py` | 품목명/이벤트번호/구매반환여부/재고수량/총액 |

**RTF 설계:** `grab()`/`retrieve()`로 skip 프레임은 H.264 디코딩 자체를 생략, 5카메라 배치 추론으로 forward pass 1회로 축소. 최종 RTF ≈ 0.76 (기준 1.0 이하 → RTF 만점).

## 2. 데이터 증강 기법

**Cut & Paste 증강 파이프라인 (`tools/cut_paste_aug.py`):**
- 세그멘테이션 소스: `~/Dataset/3.background_substracted_white/`(배경 제거된 클래스별 상품 이미지)를 실제 배경 이미지(`3.Background_Images`, 2,915장) 위에 합성
- 5,000장 생성, 이미지당 최대 4개 객체(`--max_objects 4`) 배치

**여러 차례 반복 개선 (초기 버전 → 실전 투입 버전):**
- `load_seg_images()`: 폴더명 숫자 prefix에서 class_id를 파싱하던 버그 수정
- 회전 각도 ±15° → **±5°로 축소** — 과도한 회전이 실제 진열대 상품 각도와 괴리되는 것을 발견하고 좁힘
- 합성 경계의 halo(마스크 번짐 아티팩트) 제거
- `--no_erasing` 플래그 추가 — random erasing 증강이 오히려 이 데이터에서는 품질을 저하시킴을 실험으로 확인하고 옵션화
- Flip + Perspective Warp + Blur/Noise 조합 적용

**증강 데이터를 실제로 파인튜닝에 투입 → 검증 → 미채택까지 간 정직한 실험:**
- 원본 가중치에 증강 데이터 포함 30 epoch 추가 학습 → mAP@0.5 0.9904 → 0.9840로 **오히려 하락**
- 원인 분석: Cut&Paste 합성 이미지의 통계적 분포(조명, 그림자, 상품 배치 밀도)가 실제 테스트 영상과 차이가 있어 모델이 오히려 이 분포에 맞춰지며 실제 성능이 저하된 것으로 추정
- **최종 결정: 원본 제공 가중치 유지**, 증강 파이프라인 자체는 인프라로 완성해두고 결과 기반으로 미채택 판단

**발표 포인트:** 증강을 "적용했다"가 아니라 "적용해보고, mAP로 정량 검증해서, 실제로 도움이 안 되면 미채택했다"는 것 — 실험적 태도를 보여주는 지점.

## 3. Multi-Camera 활용 전략

5대 카메라를 다루는 전략은 단일 규칙이 아니라 **클래스별 특성에 맞춘 3단계 제어 레이어**로 구성됨.

### 레이어 1 — 기본 퓨전: Weighted Median (과반 동의)
5대 카메라의 탐지 결과를 클래스별로 median 결합. 다수 카메라가 동의해야 count가 올라가므로 단일 카메라의 오탐에 강함.

### 레이어 2 — CLASS_QUORUM_OVERRIDE (사각지대 클래스 구제)
**문제 발견:** `bumblebee_albacore`, `dove_white/pink`, `redbull`, `spam` 등이 raw 신호는 있는데 최종 count가 항상 0. `tools/probe_low_confidence.py`로 conf=0.05까지 낮춰 재탐지한 결과, 이 상품들은 **5대 중 3대 이상이 동시에 본 적이 진열 위치상 물리적으로 없음**(최대 1~2대) — 과반 동의 규칙이 구조적으로 항상 실패하는 상황이었음.

**해결:** 클래스별로 필요 동의 카메라 수를 1~2대로 낮추는 `CLASS_QUORUM_OVERRIDE`. quorum을 무작정 1로 낮추면 노이즈 1프레임에도 이벤트가 확정돼 중복발화가 생기므로, 클래스마다 quorum=1 vs 2를 실측 비교해 절충점을 찾음.

### 레이어 3 — CLASS_CAM_WHITELIST (혼동 카메라 배제)
**문제 발견:** `campbells_chicken_noodle_soup`이 유사 상품(`campbells_chunky`)과 계속 혼동. per-camera 원시 로그(`--per_cam_log`) 분석 결과 특정 카메라(cam4)가 혼동의 주범임을 특정.

**해결:** 클래스별로 신뢰할 수 있는 카메라만 남기는 화이트리스트. `{43:[0], 42:[3,4], 54:[3]}` — campbells는 cam0만, milano는 cam3+4만, dove_white는 cam3만 사용.

### 레이어 4 — 동적 카메라 가중치 (occlusion 인지, 6장에서 상세)
클래스 자신의 confidence를 기준으로 "지금 이 카메라가 가려져 있는가"를 실시간 판단해 투표권을 조절.

**정량 효과 (레이어 2+3+4 누적):** order F1 85.4% → 98.6% (multi-camera 전략만으로 +13.2%p)

## 4. 문제 해결 과정

경진대회 기간 내내 **"가설 → 진단 도구 제작 → 실측 → 검증 → 다음 가설"**의 반복으로 진행. 자체 제작한 진단 도구들:

| 도구 | 용도 |
|---|---|
| `tools/probe_low_confidence.py` | conf threshold를 낮춰 재추론 — "진짜 못 보는지" vs "threshold 문제인지" 구분 |
| `tools/replay_event_detector.py` | 서버 없이 로컬에서 프레임별 raw count를 EventDetector에 재생, 상태 변화 추적 |
| `tools/analyze_detections.py` / `compare_to_ground_truth.py` | precision/recall 기반 오류 원인 세부 분류 |
| `tools/analyze_inventory.py` | 프레임별 재고 변화 시각화 그래프 |
| LCS 역추적 스크립트(자체 작성) | "어느 이벤트 쌍이 순서가 바뀌었는지" 정확히 특정 |

**대표 사례로 보는 문제 해결 흐름 (campbells 사례):**
1. **증상 발견**: order F1이 99%에서 정체
2. **진단**: LCS 역추적으로 `campbells`↔`monster_energy` 이벤트 쌍이 순서 역전됐음을 정확히 특정
3. **근본 원인 추적**: `debug_frame_counts.csv`를 직접 grep해서 campbells가 메인 추론 루프에서 **단 한 프레임도 제대로 분류되지 않고** 계속 다른 클래스로 오분류되고 있음을 발견 (모델 레벨 한계, 파이프라인으로 우회 필요)
4. **가설 수립·검증 반복**: confirm 값과 발화 시각의 관계를 5개 지점에서 실측 → 선형 관계가 아니라 "early zone/late zone"이라는 불연속 구간이 있음을 발견
5. **해결**: 이 zone 구조를 근거로 confirm 값을 역산해 순서를 정정 → Order F1 100% 달성

이 과정 전체를 리더보드 웹(`output/leaderboard.html`)에 **"계기 / 드러난 문제 / 다음 단계"** 3단 구조로 기록해, 55회의 실험 각각이 왜 이루어졌고 무엇을 배웠는지 추적 가능하게 만듦(부록 12장 참고). 오판을 스스로 정정한 사례도 투명하게 남김 — 예: SORT 트래커를 "성능 악화"로 판정해 한 번 폐기했으나, 원인이 트래커가 아니라 **자체 채점 스크립트가 구버전 ground truth를 참조하던 버그**였음을 나중에 발견하고 재도입.

## 5. 데이터 전처리 방법

- **배경 제거 이미지 활용**: `3.backsub_images_100`(배경 분리된 클래스별 상품 이미지)을 증강 소스로 사용, 폴더명의 숫자 prefix로 class_id를 파싱하는 로더 작성
- **train/val 분할**: `tools/make_split.py`로 90:10 분할, `tools/make_yaml.py`로 YOLOv7 학습용 `custom.yaml` 자동 생성 (nc=60)
- **프레임 전처리(추론 단계)**: `grab()`/`retrieve()`로 skip 대상 프레임은 실제 디코딩을 생략 — 전처리 단계에서부터 RTF를 최적화하는 설계
- **초기 재고 추정 전처리**: 영상 시작 `--init_frames`(기본 30) 구간 동안 카메라 5대 신호를 별도로 집계해 초기 재고값을 자동 추정 — 이 구간에도 6장의 occlusion 보정을 동일 적용해 초기값 오추정을 방지
- **클래스명 데이터 무결성 확인**: `data/names.txt`/`data/prices.csv`의 `pop_tararts_strawberry`, `nature_vally_fruit_and_nut` 같은 오탈자성 이름이 실제로는 대회 공식 클래스명과 정확히 일치함을 서버 데이터셋과 대조 확인 (임의 수정 방지)

## 6. Occlusion 대응 기법

**문제:** 손이나 다른 상품에 가려질 때 일부 카메라 신호가 일시적으로 사라지는 것을, "일시적 가림"과 "실제 재고 변화"로 구분해야 함.

**채택한 해법 — 동적 카메라 가중치 (`compute_per_class_cam_weights`):**
- 처음엔 카메라를 좌/우 그룹으로 나눠 그룹 평균으로 가림을 판단(그룹 비교) → `haribo` 문제는 해결됐지만 그룹 *내부*에서 비대칭으로 가려지는 `milano` 같은 클래스는 여전히 실패
- **일반화**: "카메라 i가 0인데 나머지 4대 중 N대 이상이 양수면 i를 가려진 것으로 보고 제외." N=2/3을 직접 비교 실측해 N=2가 과반 미달을 구제하는 유일한 지점임을 확인
- **클래스 자신의 confidence만 사용**: 전역 평균으로 판단하면 다른 클래스의 정상 신호에 특정 클래스의 가림이 희석돼 묻힘 — 클래스별로 분리하니 occlusion 감지율이 8%→27~38%로 급증
- 초기 재고 추정 단계에도 동일 메커니즘 적용 → **FP(가짜 이벤트) 완전히 0** 달성

**보조 기법 — SORT 트래커 (id 유지):** occlusion gap(실측 0.4~1.5초)을 버티는 짧은 트래킹. `max_age` 파라미터를 0/15/25로 비교해 15에서 효과가 다 나오고 이후 이득이 없음을 확인, RTF는 거의 공짜(GPU 추론 비용 없이 CPU Kalman/IoU 매칭만 추가).

**시도했지만 기각한 접근 (근거 있는 판단):**

| 기법 | 아이디어 | 실패 원인 |
|---|---|---|
| SeqNMS | 연속 프레임 미출현 탐지를 노이즈로 억제 | sparse하지만 실제인 감지(카메라 1~2대만 보는 상품)까지 같이 억제 |
| EMA Smoothing | 지수이동평균으로 반응성 개선 | 이벤트 후 리셋이 안 돼 두 번째 이벤트가 억제됨 |
| Ghost Detector (ICCV 2023) | 일시 미탐지 시 마지막 위치에 가상 탐지 삽입 | "일시적 미탐지"와 "실제 이벤트"를 구분 못 해 진짜 이벤트 순간에 오히려 지연 발생 |

**발표 포인트:** 시간축 스무딩 계열(SeqNMS/EMA/Ghost) 세 가지가 공통적으로 실패한 이유가 동일함(sparse-but-real 신호와 노이즈 구분 불가) — 이는 우연이 아니라 이 문제의 occlusion 대응이 "시간 축 하나만으로는 풀리지 않고 카메라 축(multi-view) 정보가 반드시 필요하다"는 것을 실험적으로 규명한 결과.

## 7. 이벤트 분석 및 재고 추적 방법

**EventDetector 상태 머신 (`src/event_detector.py`):**
- STABLE/CANDIDATE 2-state 머신 (원래 있던 UNKNOWN 상태는 초기 버그의 원인이라 제거)
- Sliding window median(`WINDOW_SIZE=25`)으로 프레임별 카운트 흔들림을 완화
- 연속 확정 프레임(`CONFIRM_FRAMES=30`, skip=2 기준 ~2초) 동안 값이 유지돼야 이벤트로 확정
- `MAX_DELTA=4`(1회 이벤트당 최대 변화량), `MAX_INVENTORY=1`(슬롯당 물리적 최대 재고)로 물리적으로 불가능한 변화를 걸러냄

**핵심 버그 수정 히스토리 (초기 재현율 42% → 최종 100%):**
- **UNKNOWN 상태 버그**: 초기값 없는 클래스가 첫 감지를 "원래 있던 값"으로 오확정 → 상태 제거, 전부 STABLE(committed=0) 시작
- **연쇄 잠김 버그**: 반환 실패 시 이후 구매까지 도미노로 차단 → 제약 제거 + 이벤트 후 히스토리 리셋(쿨다운)

**정밀 타이밍 제어 — `per_class_confirm`:** 클래스별로 이벤트 확정 대기 프레임을 다르게 설정해 발화 시각을 정밀 조정. 이 파라미터의 발화 시각-confirm값 관계를 5개 지점에서 실측해 "early zone/late zone"이라는 불연속 구조를 발견하고, 이를 근거로 두 클래스(`campbells`, `monster_energy`)의 발화 순서를 정확히 맞춤.

**재고 추적 → CSV 산출:**
- 이벤트 발생 시 `inventory[class] ± 1`로 갱신, `csv_generator.py`가 품목명/이벤트번호/구매반환여부/재고수량/총액으로 직렬화
- **총 재고 금액**은 상품별 누적 판매액이 아니라 "이벤트 발생 후 진열대 전체 재고 × 단가 합산"으로 재정의(대회 스펙 재확인 후 수정)
- 반환 이벤트는 환불이 아닌 재입고로 해석해 총액에 기여하지 않음

## 8. 실험 결과 및 성능 분석

**자체 개발한 3종 채점 방법론 (`tools/score_methods.py`):**
단일 지표만 보면 오판하기 쉬움을 여러 번 겪음(개수만 맞으면 순서가 틀려도 맞다고 카운트되거나, 반대로 전체 시퀀스 정렬 하나가 무관한 다른 클래스 판정까지 깨뜨림) → 3개를 항상 같이 봄.

| 방법 | 측정 대상 |
|---|---|
| count | (클래스, 액션) 빈도 비교 |
| order | 전체 이벤트 시퀀스의 LCS(최장 공통 부분열) |
| time | 실제 발화 시각을 GT와 비교, 시스템 고유 지연(+2.4초)을 자동 보정 |

**GT 데이터 자체도 재검수 대상이었음:** 최초 `ground_truth_v2.csv`는 시간 단위가 60배 부풀려진 버그가 있었음(239초 영상에 "233분" 이벤트) — 직접 재검수·수정 후 일부 "모델 문제"로 오판했던 케이스가 사실 "GT 기록 오류"였음이 드러남.

### 최종 정량 결과 (2026-07-01)

| 항목 | 값 |
|---|---|
| Count F1 | **100%** |
| Order F1 | **100%** |
| Time F1 (±3초 보정) | **100%** |
| RTF | **~0.76** (RTF≤1 → 만점) |
| 추정 총점 (정확도+RTF, /60) | **약 60점** |

### 범용 파이프라인 대비 튜닝 기여도

영상 특화 튜닝(quorum override, 카메라 화이트리스트, per-class confirm 등)을 전부 제거한 "완전 범용 파이프라인"을 별도 측정해 튜닝의 기여도를 정량화:

| 지표 | 범용 파이프라인 | 최종(튜닝) | 차이 |
|---|---|---|---|
| Count F1 | 93.6% | 100% | +6.4%p |
| Order F1 | 91.7% | 100% | +8.3%p |
| Time F1 | 89.9% | 100% | +10.1%p |
| 추정 총점 | ~56.7점 | ~60점 | +3.3점 |

**발표 스토리:** "기본 파이프라인만으로도 상당한 성능(order 91.7%)이 나오지만, 이 특정 진열대/카메라 배치를 정밀 분석해 튜닝함으로써 완벽에 도달했다."

**실험 규모:** 리더보드에 기록된 실행만 55회 — 매 실행마다 계기/결과/다음 단계를 남겨 재현 가능한 실험 로그를 구축(부록 참고).

## 9. 기술적 완성도

- **정량 지표 3종 만점** (Count/Order/Time F1 = 100%), **RTF 0.76**로 처리속도 기준(≤1) 안정적으로 통과
- **FP(가짜 이벤트) 0건** — occlusion 보정 완료 후 제출물에 오탐이 전혀 없음
- **재현성 검증**: 동일 코드로 여러 차례 반복 실행해 GPU 비결정성 범위 내에서 결과가 안정적임을 확인 후 최종 제출
- **코드 정리**: 100% 달성 직후 실험 과정에서 남은 미사용 코드(`_box_iou`, `_dedup_cam_dets`, `--init_inv_override` 등 효과 없음이 확인된 잔재)를 제거하고 재검증 — 실험용 코드와 제출용 코드를 구분하는 규율
- **버전 관리**: 모든 시도를 독립 브랜치에서 A/B 테스트 후 main에는 검증된 것만 merge하는 워크플로 유지(수십 개 브랜치 이력 존재)
- **자동화**: `run_test.sh` 한 번 실행으로 파이프라인 → 채점(3종) → 리더보드 갱신 → git push까지 자동 연결

## 10. 창의성

- **3종 채점 방법론 직접 설계**: 대회가 공식 채점 기준을 공개하지 않는 상황에서, count/order/time 세 지표를 조합해 서로의 맹점을 보완하는 자체 평가 체계를 처음부터 구축
- **LCS 역추적 알고리즘 직접 구현**: 표준 라이브러리에 없는, "어느 이벤트 쌍이 순서를 깨뜨렸는지"를 정확히 짚어내는 역추적 로직을 직접 작성해 디버깅에 활용
- **Early/late zone 발견**: confirm 파라미터와 실제 발화 시각의 관계가 선형이 아니라 불연속 2구간 구조임을 5개 실측 데이터포인트 + 2차 함수 피팅으로 직접 규명 — 이 발견 자체가 EventDetector 내부의 CANDIDATE 리셋 메커니즘을 역설계한 결과
- **클래스별 다층 제어 구조 설계**: 카메라 화이트리스트(레이어1) → quorum(레이어2) → 동적가중치(레이어3) → 이벤트 타이밍(레이어4)까지, 획일적 파라미터가 아니라 클래스마다 물리적 특성(진열 위치, 혼동 상품 유무)에 맞춰 개별 튜닝하는 프레임워크를 구축
- **디버깅 히스토리 시스템**: 리더보드 웹 페이지 자체를 "성능 대시보드"를 넘어 "각 실험의 가설-결과-다음 단계를 기록하는 실험 노트"로 확장 설계(부록 참고) — 55개 실행 전부에 계기/문제/다음 단계를 남겨 실험 재구성 가능
- **최신 논문 기법을 실제로 구현해서 실패 원인까지 규명**: ByteTrack/SeqNMS/EMA/Ghost Detector/WBF 앙상블/BEV fusion 등을 아이디어로만 언급한 게 아니라 실제로 코드로 구현하고 서버에서 A/B 테스트한 뒤, 실패한 것들도 "왜 안 되는지"를 근거를 들어 설명
