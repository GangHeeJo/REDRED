00 · Background

Five diagnostic tools we built before touching the model

The competition never published an official scoring formula. Every diagnostic and automation tool below was built from scratch, and every one of them gets reused across both architectures later in this deck.

✓
3-way scoring methodology (count / order / time, tools/score_methods.py)

Count alone can be "correct" even when order is wrong, and a single misordered pair can wrongly fail unrelated classes — the three metrics cover each other's blind spots.

✓
LCS backtracking algorithm

When Order F1 shows a mismatch, this pinpoints exactly which pair of events swapped places — down to the index.

✓
Experiment-log leaderboard (output/leaderboard.html)

Every one of 55+ runs is logged with what motivated it, what problem it exposed, and what came next — an expandable lab notebook, not just a scoreboard.

✓
debug_frame_counts.csv replay tool

Replays raw per-frame counts locally, no server needed, to separate "the model genuinely missed it" from "the pipeline logic is buggy."

✓
Candidate-time back-calculation (fire_time formula)

Works backward from an event's confirmed firing time to tune confirm_frames precisely — reused verbatim in both the YOLOv7 and RF-DETR sessions.

01 · YOLOv7

Self fine-tuning fails → pivot to pipeline engineering

Starting point: the competition-provided weights (yolov7_custom.pt, mAP@0.5 98.1%, trained on roughly 200,000 images).

The pivot — the lighting, shadow, and object-placement distribution of the augmented images diverged from the real test footage, so the model drifted toward that synthetic distribution instead of improving. Conclusion: touching the model itself is risky. Keep the provided weights untouched, and improve the pipeline that sits on top of them instead.

02 · YOLOv7 — Full Pipeline

The YOLOv7 pipeline: four layers on top of a frozen model

Because the detector weights are frozen, every gain from here on comes from four layers of pipeline logic sitting above it — each one solving a problem the last one couldn't.

LAYER 1

Multi-Camera Fusion

Weighted median + per-class quorum override + camera whitelist

LAYER 2

Occlusion-Aware Weighting

Global → per-class → per-camera exclusion, 3 stages of refinement

LAYER 3

SORT Tracker

max_age=15, bridges short occlusion gaps

LAYER 4

Hysteresis State Machine

STABLE/CANDIDATE + per-class, per-direction confirm timing

Why four layers — each layer targets a failure mode the previous one leaves behind: physical blind spots (L1), transient hand occlusion (L2), short tracking gaps (L3), and precise event-timing disputes between classes (L4). The next three slides go through each in turn.

Layer 1 — Multi-Camera Fusion

1

Quorum override + camera whitelist

Default rule: weighted median across 5 cameras (majority agreement). Two class-specific exceptions sit on top of that default.

Q

CLASS_QUORUM_OVERRIDE

Some products are physically visible from only 1-2 of the 5 cameras due to shelf placement — a structural blind spot no "majority rule" can satisfy. For those classes (8 in total), the required agreement count is relaxed to 1-2 cameras.

W

CLASS_CAM_WHITELIST

Per-camera logs showed specific cameras (e.g. one angle confusing campbells with milano or dove_white) were the source of repeated misclassification — those cameras are excluded outright for the affected classes.

03 · The YOLO11 Attempt

YOLO11 attempt: higher mAP, worse real-world accuracy

Hypothesis: a newer, lighter architecture (YOLO11) should beat the competition weights on both accuracy and speed.

AttemptResult
YOLO11m retrained (18,392 images, 50 epochs)mAP 98.4% (comparable/higher) · order F1 73.4% (−23.2pp)
YOLO11m + Knowledge Distillationorder F1 91.1% (−7.5pp vs. YOLOv7)
The real cause wasn't model architecture — it was raw data volume — the provided YOLOv7 weights were trained on roughly 200,000 images; the team's own retraining set had only 18,000 (a 10x+ gap). mAP on a standard benchmark can look similar even when the model has nowhere near enough coverage of the real footage's angles, lighting, and occlusion patterns.
This realization set the strategy going forward — "the next time we try a different model, the question isn't just architecture — it's how we secure enough raw data."

05 · Adopting the Full RF-DETR Stack

First attempt: reusing YOLOv7's code plateaus at 85.6%

Developed on a separate branch (feat/rfdetr-sam2) — main and the final submission were never touched. Adopting RF-DETR meant rebuilding the entire stack, from preprocessing to decision logic.

1

YOLOv7 code ported as-is

First scored with run_pipeline_rfdetr.py, which reused event_detector.py/multi_view_fusion.py unchanged — 10-epoch order F1 85.2%

2

50-epoch retrain (no augmentation) → got worse

Order F1 dropped to 82.6% — 9 new classes started over-firing. Confirmed for the second time in this project that higher mAP doesn't translate into better pipeline performance (same lesson as YOLO11).

3

RTF retuning + accumulated fixes → order F1 85.6%

Adjusted skip=3/conf=0.35, added CLASS_CONF_OVERRIDE plus per-class whitelist/quorum, following the same playbook as the YOLOv7 session

Copying YOLOv7's diagnostics and tuning stack wasn't enough to reach 100% — it became clear that two things were needed at once: (1) more raw data, and (2) a pipeline redesigned around RF-DETR's own characteristics.

06 · Diagnosis

Diagnosis: the real problem was missing training data

Breaking down the 85.6% plateau class by class.

Symptom — specific classes (campbells, hunts_sauce, cheerios, and others) kept failing to be detected in a structural way.
Diagnosis — the same pattern as YOLO11. Not a model-architecture problem, not a pipeline-logic problem — the training data simply didn't contain enough exposure to those specific classes.
Conclusion — rather than tune more parameters, expanding raw data is the root-level fix. The competition only ever handed over the weights trained on 200,000 images — never the original images themselves — so growing the dataset had to be done in-house.

07 · Expanding Raw Data With SAM2

SAM2: extracting training data straight from the test video

The competition handed over trained weights, not the 200,000 images behind them. So instead of chasing that number, the team pointed SAM2 at the one thing it actually had unlimited access to: the competition's own test footage.

Team dataset

~20,000 images owned by the team

COCO conversion

tools/yolo_to_coco.py — full set, no filtering

SAM2 domain extraction

tools/sam2_video_label.py — labels the competition's own test footage frame by frame

Weak/timing-class boost

3-tier capture threshold + 5x oversampling

Retrain (30 epochs)

mAP 0.898 → 0.9045 at epoch 14

1

All ~20,000 team images converted to COCO format

The full set, not a hand-picked subset — there was no "select the 200k-equivalent images" step, because the 200k images were never in the team's hands to begin with.

2

SAM2 turns the competition's own test video into frames

Extracts and labels additional domain data directly from the target footage — camera angle, lighting, and occlusion patterns match the actual scoring video by construction, closing the train/test distribution gap that no amount of generic augmentation could.

3

A 3-tier capture threshold, not a blanket lower bar

Lowering SAM2's confidence threshold everywhere risks flooding good classes with noisy labels — so the threshold was lowered only for the classes that actually needed it, in two different degrees.

07 · Pipeline Redesign — The Probabilistic Core

One formula standing in for four layers

The redesign starts from a fact the YOLOv7 session had already proven: across every one of the 105 ground-truth events, inventory for any single class is never anything but 0 or 1. Counting was never actually the problem — presence was.

P = 1 − ∏ (1 − confi)
Every active camera i contributes independent evidence for "this class is present." Combining them as a probability means one confident camera can outweigh several uncertain ones — the decision no longer depends on how many cameras agree, only on how convincing the evidence is in total.

YOLOv7 — quorum voting

needs N cameras ≥ threshold

A hard camera-count rule. Confidence values themselves are never trusted directly — they only gate whether a camera's vote counts at all, so vote-counting logic (whitelist, quorum override, occlusion exclusion) has to do all the work.

vs

RF-DETR — Noisy-OR fusion

P ≥ 0.7 combined probability

Two cameras independently reporting 0.5 confidence combine to P = 1−(0.5×0.5) = 0.75 — already past the 0.7 default threshold, without ever having to define "how many cameras counts as a quorum" for that class.

Why this only works for RF-DETR — Noisy-OR treats each confidence score as a real probability. That's only valid if the detector's confidence is actually calibrated (a 0.7 really does mean "roughly 70% likely to be right"). YOLOv7's confidence couldn't be trusted at that level — which is exactly why it needed four layers of workaround instead of one formula. More on this in the confidence-calibration slide.

07 · Pipeline Redesign — Configuration Architecture

Every parameter, per class — and zero code changes to retune

All class-specific behavior lives in one file, config/rfdetr_native_class_config_v2_reinforced.json, loaded into a ClassConfig object with 12 independently tunable dimensions. Where YOLOv7's overrides are Python dicts baked into multi_view_fusion.py — edit, commit, push, pull, rerun — retuning RF-DETR is a JSON edit and nothing else.

Camera selection

whitelist — restrict which cameras vote for a class

quorum / conf — per-class agreement count and effective confidence floor

Event timing

confirm_frames — either one number, or {"0": purchase, "1": return} when the two directions need opposite tuning

refractory_frames, min/max_confirm_frames + adaptive_confirm_classes (opt-in signal-strength interpolation)

Fusion sensitivity

presence_threshold — per-class override of the default 0.7 Noisy-OR cutoff

duplicate_penalty — e.g. milano: 1.0 disables the default 0.5x penalty for classes where duplicate boxes are a real detection, not noise

Signal shaping

window_size — sliding-window length for the presence vote

low_thresh — how easily a class is allowed to register as "gone"

Example — milano (class 42) — confirm_frames: 45, duplicate_penalty: 1.0. Milano genuinely produces duplicate boxes on legitimate detections, so the penalty that protects every other class from double-counting would only hurt this one — one line in the JSON turns it off, for this class alone.

10 · What This Competition Taught Us

Five lessons that outlasted any single model

Two independent 100%-score runs, on two unrelated architectures, kept teaching the same handful of lessons. These are the ones the team would carry into the next project.

1
mAP is not pipeline accuracy

Confirmed independently three separate times — YOLO11 retraining, RF-DETR's first 50-epoch pass, and knowledge distillation all raised mAP while making the actual event detection worse. This single lesson shaped every architecture decision after the first time it happened.

2
More data beats a newer architecture

Every "switch to a fancier model" attempt only started working once the data problem underneath it was fixed first — YOLO11 never got that fix and stayed broken; RF-DETR got SAM2 and reached 100%.

3
A model's confidence quality decides how much code you have to write around it

YOLOv7's four-layer correction stack and RF-DETR's single Noisy-OR formula solve the exact same problem — the difference is entirely about how much the raw confidence score could be trusted.

4
Literature-recommended tricks still need to be tested on your own data

ByteTrack, SeqNMS, WBF, EMA smoothing all have real published results behind them — and all six failed here for reasons specific to this dataset. Reading the paper was never a substitute for running the experiment.

5
A debugging method can outlive the codebase it was built for

The candidate-time back-calculation technique invented to fix one YOLOv7 ordering bug was reused, unmodified, to fix RF-DETR bugs in a completely different pipeline months later. The tooling turned out to be more durable than either model.

Conclusion

Same principle, different architecture, same 100%

Self fine-tuning failed, so the team pivoted to pipeline engineering. The YOLO11 attempt hit a raw-data wall, and that lesson carried straight into RF-DETR — apply the same principle ("expand raw data directly, from the target footage itself") and reach the same 100% with a shallower, more principled pipeline.
